1 Présentation du projet

Le Chatbot EDT de l'UGANC est une application web développée avec Laravel 10 (PHP) permettant à l'ensemble de la communauté universitaire — étudiants, enseignants et chefs de programme — de consulter, gérer et recevoir des alertes sur l'emploi du temps de l'université.

L'application couvre trois rôles distincts :

  • Chef de programme : administre l'emploi du temps de son département, publie des annonces et fait progresser les étudiants d'une année à l'autre.
  • Professeur (prof) : consulte ses cours assignés et signale les modifications d'horaires.
  • Étudiant : consulte son emploi du temps, reçoit des alertes de modification et des notifications en temps réel.
Périmètre technique : Laravel 10 · MySQL · API Groq (LLaMA 3.1) · API Tavily (recherche web) · JavaScript natif · Blade Templates · Bootstrap / CSS personnalisé.

2 Problématique : pourquoi intégrer une IA ?

Sans intelligence artificielle, l'application est fonctionnelle mais rigide : l'utilisateur doit naviguer manuellement dans des menus pour trouver une information. L'objectif était de rendre l'application conversationnelle, c'est-à-dire capable de :

  • Répondre en langage naturel (français ou anglais) aux questions des utilisateurs ;
  • Accéder en temps réel à toutes les données de la base de données ;
  • Déclencher automatiquement les vues de l'interface (tableaux, alertes, PDF) selon la question posée ;
  • Répondre aux questions générales sur les universités, la Guinée et le monde entier ;
  • Effectuer des recherches sur internet pour des informations récentes.
Exemple concret : Au lieu de cliquer sur « Filière → Niveau → Lundi », l'utilisateur écrit simplement « Qui donne cours lundi matin ? » et l'IA affiche automatiquement le tableau de l'EDT du lundi.

3 Choix du modèle de langage (LLM)

3.1 Analyse des options disponibles

Plusieurs options ont été évaluées avant de choisir la solution finale :

Solution Avantages Inconvénients Verdict
Google Gemini Gratuit, API simple Quota = 0 sur le compte testé, clé invalide ❌ Abandonné
OpenAI GPT-4 Très performant Payant, pas accessible sans carte ❌ Impossible
Ollama (local) Totalement gratuit Nécessite un serveur puissant ❌ Infrastructure indisponible
Groq API Gratuit, très rapide, compatible OpenAI Limite de tokens par minute (6 000 TPM) ✅ Retenu

3.2 Pourquoi Groq ?

Groq est une plateforme cloud qui héberge des modèles de langage open-source et les exécute sur des puces spécialisées (Language Processing Units), ce qui les rend extrêmement rapides. L'offre gratuite permet :

  • 14 400 requêtes par jour (largement suffisant pour un usage académique) ;
  • 6 000 tokens par minute (limite principale à gérer) ;
  • Une API au format identique à OpenAI (facilité d'intégration) ;
  • Le modèle LLaMA 3.1 8B Instant : rapide, multilingue, performant.
Modèle utilisé : llama-3.1-8b-instant
Développé par Meta (Facebook AI Research), entraîné sur des milliards de textes multilingues, capable de comprendre le français, l'anglais et de raisonner sur des données structurées.

5 Le contrôleur IA : ChatbotAIController

5.1 Fichier et rôle

Tout le cerveau de l'IA est centralisé dans un seul fichier Laravel :

app/Http/Controllers/ChatbotAIController.php

Ce contrôleur est appelé via la route POST /api/chatbot/ai et orchestre toutes les opérations.

5.2 Méthode principale : ask()

La méthode ask() reçoit la requête HTTP et effectue les étapes suivantes dans l'ordre :

  1. Récupération des paramètres : message de l'utilisateur, historique de conversation, identifiants de filière, niveau et enseignant.
  2. Identification de l'utilisateur connecté : si un utilisateur est authentifié, ses données (filière, niveau, rôle) sont automatiquement utilisées.
  3. Collecte des données en base : 8 appels à la base de données en parallèle (EDT, alertes, annonces, filières, niveaux, statistiques, enseignants, étudiants, chefs).
  4. Décision de recherche web : si la question ne concerne pas l'application, une recherche Tavily est lancée.
  5. Construction du prompt système : toutes les données sont intégrées dans un texte d'instruction structuré.
  6. Appel à l'API Groq : envoi du prompt + historique + question au modèle LLaMA.
  7. Retour JSON : { "message": "...", "action": "...", "params": {...} }.
// Extrait simplifié de la méthode ask()
public function ask(Request $request)
{
    $message   = $request->input('message');
    $history   = $request->input('history', []);
    $filiereId = $request->input('filiere_id');

    // 1. Collecte des données de la BD
    $edt         = $this->fetchEdt($filiereId, $niveauId, $enseignantId);
    $enseignants = $this->fetchEnseignants($filiereId);
    $stats       = $this->fetchStats();
    // ... (8 méthodes de collecte au total)

    // 2. Recherche web si nécessaire
    $webResults = $this->needsWebSearch($message)
        ? $this->searchWeb($message)
        : null;

    // 3. Prompt système complet
    $systemPrompt = $this->buildSystemPrompt(..., $webResults);

    // 4. Appel Groq (format OpenAI)
    $res = Http::post('https://api.groq.com/openai/v1/chat/completions', [
        'model'    => 'llama-3.1-8b-instant',
        'messages' => $messages,
    ]);

    return response()->json(json_decode($res->json('choices.0.message.content')));
}

6 Accès complet à la base de données

6.1 Principe

L'une des fonctionnalités les plus importantes est que l'IA a accès en lecture à l'intégralité de la base de données à chaque requête. Cela signifie que l'IA ne travaille jamais avec des données statiques ou anciennes : elle reçoit les informations les plus récentes à chaque message.

6.2 Les 8 méthodes de collecte

MéthodeDonnées collectéesFiltrage
fetchEdt() Emplois du temps (jour, heure, matière, enseignant, salle) Par filière + niveau + enseignant
fetchAlertes() Modifications d'horaires (72 dernières heures) Par filière
fetchAnnonces() Annonces du département Par filière (+ annonces globales)
fetchFilieres() Liste de tous les départements (id, nom, code) Aucun (toutes les filières)
fetchNiveaux() Niveaux académiques (L1, L2, L3…) avec leur filière Aucun (tous les niveaux)
fetchStats() Compteurs globaux : nb étudiants, profs, cours, filières Aucun
fetchEnseignants() Liste complète des enseignants (nom, prénom) Par filière si précisé
fetchEtudiants() Liste des étudiants avec leur filière et niveau Par filière
fetchChefs() Chefs de programme et leur département Aucun
Résultat : À chaque message de l'utilisateur, l'IA reçoit une "photographie" complète et fraîche de la base de données. Si un chef modifie un horaire à 14h00 et qu'un étudiant pose une question à 14h01, l'IA donnera déjà la bonne réponse avec le nouvel horaire.

7 Le prompt système : cerveau de l'IA

7.1 Qu'est-ce qu'un prompt système ?

Un prompt système (ou system prompt) est un texte d'instructions que l'on envoie au modèle de langage avant le message de l'utilisateur. Il définit le comportement, le rôle, les règles et les données disponibles pour l'IA. C'est l'équivalent d'un "contrat de travail" pour l'IA.

7.2 Structure du prompt système

Le prompt est généré dynamiquement par la méthode buildSystemPrompt() et contient les sections suivantes :

  1. Identité et langue : "Tu es l'assistant IA du système EDT de l'UGANC. Réponds dans la même langue que l'utilisateur."
  2. Contexte temporel : date actuelle, jour de la semaine, heure exacte. Cela permet à l'IA de répondre "Qui enseigne aujourd'hui ?" correctement.
  3. Profil utilisateur connecté : nom, rôle (chef/prof/étudiant), département, niveau. L'IA sait à qui elle parle.
  4. Données de la base de données : filières, niveaux, emploi du temps complet, alertes, enseignants, étudiants, chefs, statistiques — tout en format compact.
  5. Résultats web (si applicable) : informations trouvées sur internet via Tavily pour les questions mondiales.
  6. Liste des actions disponibles : l'IA peut déclencher des actions dans l'interface (voir section 10).
  7. Règles de comportement : 9 règles précises pour éviter les hallucinations et garantir des réponses pertinentes.

7.3 Les 9 règles comportementales

Ces règles ont été affinées au fil des tests pour corriger les problèmes rencontrés :

#RègleProblème corrigé
1 Questions sur l'application → données BD uniquement Empêche l'IA d'inventer des horaires
2 Questions générales → connaissances + résultats web Permettre les questions sur la Guinée, universités, etc.
3 Convertir les noms en IDs dans les params des actions Permet à l'interface d'exécuter l'action directement
4 "Qui enseigne aujourd'hui ?" → consulter l'heure actuelle Réponse contextualisée selon le moment de la journée
5 Ne jamais inventer de données spécifiques à l'app Correction de l'hallucination "Sophie connectée"
6 Rôles : chef=responsable, prof=enseignant, étudiant=élève L'IA comprend la hiérarchie universitaire
7 "Voir tout ensemble" → action show_edt_jour Correction de la réponse "diplopie" hors contexte
8 Si la conv. portait sur les horaires, rester dans ce contexte Cohérence de la conversation
9 Jamais de réponse médicale/hors-sujet dans contexte EDT Correction des hallucinations thématiques

8 Optimisation des tokens

8.1 Problème initial

Lors des premiers tests, le système renvoyait une erreur "Rate limit exceeded". L'analyse a montré que le prompt système contenait environ 3 400 tokens, ce qui dépassait la limite gratuite de Groq (6 000 tokens par minute, prompt + réponse compris).

8.2 Techniques d'optimisation appliquées

Plusieurs optimisations ont été développées pour réduire la taille du prompt tout en conservant toutes les informations nécessaires :

Technique 1 — Format compact "id:nom"

Au lieu d'écrire :

Filières disponibles :
  - ID 1 : Nouvelles Technologies de l'Information et de la Communication (NTIC)
  - ID 2 : Droit et Libertés

On écrit :

DEPARTMENTS: 1:NTIC, 2:DL

Technique 2 — EDT en une ligne par cours

Format ultra-compact :

Lundi 08:00-10:00: Algorithmique | Makera Kaba | Salle A [f:1,n:2]

Technique 3 — Limitation des entrées

Maximum 20 entrées dans l'EDT, 3 annonces, 5 alertes. Les données les plus récentes ont la priorité.

Technique 4 — Historique limité

Seuls les 10 derniers échanges (20 messages) sont envoyés au modèle. Les messages trop anciens sont automatiquement supprimés.

Résultat : Le prompt est passé de ~3 400 tokens à ~800 tokens en moyenne, soit une réduction de 76 %. Zéro erreur de limite depuis l'optimisation.

9 Recherche web en temps réel (Tavily)

9.1 Pourquoi une recherche web ?

Le modèle LLaMA 3.1 a une date de coupure de connaissances (il ne connaît pas les événements très récents). De plus, pour des questions très spécifiques sur les universités guinéennes, l'actualité nationale ou des sujets pointus, il est préférable d'interroger internet directement.

L'API Tavily a été choisie car elle est spécialement conçue pour alimenter des modèles de langage avec des résultats de recherche web en temps réel. L'offre gratuite offre 1 000 recherches par mois.

9.2 Détection intelligente : faut-il chercher sur le web ?

La méthode needsWebSearch() analyse la question de l'utilisateur. Si elle contient des mots-clés liés à l'application (EDT, cours, filière, niveau, salle, enseignant, NTIC, etc.), aucune recherche web n'est effectuée — les données de la BD suffisent. Dans les autres cas, une recherche est lancée.

private function needsWebSearch(string $message): bool
{
    $appKeywords = ['edt', 'cours', 'filiere', 'niveau', 'enseignant',
                    'salle', 'ntic', 'dl', 'lundi', 'mardi', ...];

    $lower = mb_strtolower($message);
    foreach ($appKeywords as $kw) {
        if (str_contains($lower, $kw)) return false; // → BD suffisante
    }

    if (str_word_count($message) <= 3) return false; // Courte question → pas de web

    return true; // → Recherche Tavily
}

9.3 Exemples

QuestionSource utilisée
« Qui donne cours lundi ? »Base de données (EDT)
« Parle-moi de l'université de Nongo »Tavily (web)
« Combien d'étudiants sont inscrits ? »Base de données (stats)
« Quelles sont les universités de Guinée ? »Tavily (web) + connaissances LLaMA

10 Système d'actions IA → Interface

10.1 Principe

L'innovation majeure de l'architecture est que l'IA ne se contente pas de répondre en texte : elle peut aussi déclencher des actions dans l'interface graphique. Pour cela, l'IA retourne toujours sa réponse dans un format JSON structuré :

{
  "message": "Voici l'emploi du temps du lundi pour votre département.",
  "action":  "show_edt_jour",
  "params":  { "jour": "Lundi", "filiere_id": 1, "niveau_id": 3 }
}

Le frontend JavaScript intercepte ce JSON et exécute la fonction correspondante à l'action.

10.2 Liste des actions disponibles

ActionEffet dans l'interfaceParamètres
show_edt_jour Affiche le tableau de l'EDT d'un jour précis jour, filiere_id, niveau_id
show_edt_semaine Affiche l'EDT de toute la semaine courante filiere_id, niveau_id
show_edt_semestriel Affiche l'EDT complet du semestre filiere_id, niveau_id
show_alertes Affiche les modifications d'horaires récentes aucun
show_planning_enseignant Affiche le planning d'un enseignant précis enseignant_id, enseignant_nom
download_pdf Génère et télécharge le PDF de l'EDT filiere_id, niveau_id
none Réponse texte simple, aucune action visuelle aucun

10.3 Avantage clé

L'IA envoie les filiere_id et niveau_id directement dans les params. Cela signifie que l'utilisateur n'a pas besoin de sélectionner manuellement sa filière avant de poser une question : l'IA identifie elle-même le département concerné à partir des données du prompt.

Exemple : Un étudiant NTIC L2 écrit "montre moi mon emploi du temps de la semaine". L'IA retourne action: "show_edt_semaine", params: {filiere_id: 1, niveau_id: 3} et le tableau s'affiche immédiatement, sans aucun clic supplémentaire.

11 Historique de conversation

L'IA maintient un historique de conversation (mémoire à court terme) pour comprendre le contexte des messages successifs. Cela permet des échanges naturels du type :

Utilisateur : « Qui donne cours lundi ? »
IA : « Makera Kaba et Ibrahima Camara... »
Utilisateur : « Et le jeudi ? » ← L'IA comprend qu'on parle toujours des enseignants.

Techniquement, les 10 derniers échanges (20 messages au maximum) sont envoyés à l'API Groq dans le tableau messages au format standard OpenAI :

$messages = [
    ['role' => 'system',    'content' => $systemPrompt],  // Instructions
    ['role' => 'user',      'content' => 'Qui donne cours lundi ?'],
    ['role' => 'assistant', 'content' => 'Makera Kaba...'],
    ['role' => 'user',      'content' => 'Et le jeudi ?'],  // Question actuelle
];

Ce format est identique à celui utilisé par ChatGPT, ce qui démontre la compatibilité entre les différentes API de modèles de langage.

12 Notifications en temps réel

12.1 Architecture des notifications

Un système de notifications en temps réel a été développé pour informer automatiquement les utilisateurs des changements qui les concernent. Le système repose sur trois composants :

  • NotificationApp : modèle Eloquent représentant une notification (titre, message, type, destinataire, lu/non-lu).
  • NotificationService : service PHP centralisé pour créer et envoyer des notifications selon le contexte.
  • NotificationController : contrôleur API exposant les endpoints HTTP.

12.2 Types de notifications envoyées automatiquement

ÉvénementDestinatairesDéclencheur
Modification d'horaire Étudiants du département + enseignant concerné Chef de programme
Mise à jour par un professeur Étudiants du département Professeur
Progression d'année (L1 → L2) Chaque étudiant individuellement Chef de programme

12.3 Technique de polling

Les notifications sont récupérées par le frontend via un mécanisme de polling (interrogation périodique) toutes les 30 secondes :

// JavaScript — polling toutes les 30 secondes
setInterval(fetchNotifications, 30000);

async function fetchNotifications() {
    const res  = await fetch('/api/notifications');
    const data = await res.json();
    const count = data.unread_count;
    // Afficher le badge rouge sur la cloche si count > 0
}

Cette approche a été choisie pour sa simplicité et sa compatibilité universelle, sans nécessiter WebSockets ou Laravel Echo.

12.4 Scoping par département

Les notifications sont scoped par filière : un étudiant NTIC ne reçoit jamais les notifications du département DL, même si les deux filières sont dans la même base de données. Ce cloisonnement est appliqué dans NotificationService via le filiere_id.

13 Progression d'année académique

13.1 Problématique

En fin d'année, les étudiants de L1 doivent "passer" en L2. Cette opération ne peut être effectuée que par le chef de programme du département concerné et ne doit affecter que les étudiants de ce département — pas ceux d'autres filières.

13.2 Implémentation

Le ProgressionController gère cette fonctionnalité :

  • GET /api/chef/niveaux : retourne les niveaux de la filière du chef connecté avec le nombre d'étudiants par niveau.
  • POST /api/chef/progression : déplace tous les étudiants d'un niveau source vers un niveau cible.

Validations de sécurité appliquées :

  1. Vérification que l'utilisateur est bien un chef de programme.
  2. Vérification que les deux niveaux (source et cible) appartiennent à la filière du chef.
  3. Envoi d'une notification personnalisée à chaque étudiant promu.
Résultat : Si le chef de programme NTIC lance la progression L1 → L2, seuls les étudiants NTIC-L1 sont déplacés vers NTIC-L2. Les étudiants DL ne sont pas affectés.

Le développement a nécessité de résoudre plusieurs défis techniques importants :

15 Conclusion

15.1 Récapitulatif des apports de l'IA

L'intégration de l'intelligence artificielle dans le Chatbot EDT de l'UGANC a transformé une application de gestion classique en un assistant conversationnel intelligent capable de :

  • Répondre en langage naturel (français, anglais) à toute question sur l'emploi du temps ;
  • Accéder en temps réel à l'intégralité de la base de données ;
  • Déclencher automatiquement des vues de l'interface sans que l'utilisateur ait à naviguer ;
  • Rechercher des informations sur internet pour les questions mondiales ;
  • Maintenir le contexte d'une conversation sur plusieurs échanges ;
  • Respecter le cloisonnement par département (sécurité des données).

15.2 Technologies utilisées — Récapitulatif

Laravel 10 (PHP) MySQL LLaMA 3.1 8B Groq API Tavily Search API JavaScript ES6 Blade Templates OpenAI API Format Eloquent ORM Notification Polling

15.3 Perspectives d'amélioration

  • Intégration de WebSockets (Laravel Echo + Pusher) pour des notifications instantanées sans polling.
  • Passage à un modèle plus puissant (LLaMA 3.3 70B ou Mixtral) pour des réponses encore plus précises.
  • Ajout d'une synthèse vocale pour rendre le chatbot accessible aux utilisateurs malvoyants.
  • Interface mobile dédiée (application Android/iOS).
